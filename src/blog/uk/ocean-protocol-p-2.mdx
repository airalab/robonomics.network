## Вступ

У цій статті [розширюється інтеграція](/blog/article-ocean-p1/) між мережею Robonomics та протоколом Ocean. Тут буде описано конкретно, як підключити роботичні пристрої до протоколу, а також можливі варіанти використання, де така інтеграція додасть цінність до роботи автономної системи.

Для тих, хто не знайомий з Ocean Protocol: ідея полягає в перетворенні будь-яких даних на повноцінний актив шляхом токенізації, з можливістю купівлі / продажу та передачі власності. Коротко кажучи, при токенізації даних ви розгортаєте та випускаєте токен ERC20 (дататокени), власність якого дозволяє вам мати доступ до даних, призначених для нього (1 дататокен = доступ до набору даних). Протокол дозволяє гнучко керувати доступом, наприклад, організовувати тимчасовий доступ до даних або надавати доступ до їх аналітики без порушення конфіденційності. Таким чином, стає можливим монетизувати токенізовані дані, створювати для них ринки та використовувати повний потенціал інструментів децентралізованої фінансової сфери.

Команда Robonomics добре усвідомлює цінність монетизації через токенізацію, але ще краще ми розуміємо цінність автоматизації. Ми вже вивчили, як автоматично монетизувати корисні дії роботів та пристроїв Інтернету речей, тому чому б не автоматизувати монетизацію результатів (журналів та даних) цієї діяльності? З допомогою Ocean Protocol майже будь-який роботичний пристрій, який генерує цінні дані, може бути перетворений на просте джерело доходу для його власника. Або пристрій може стати споживачем корисних даних для завершення своєї задачі.

## Аспекти інтеграції для пристроїв ROS

Для того щоб Robonomics працював з багатьма різними типами пристроїв, ми використовуємо Robot Operating System, набір програмного забезпечення, який дозволяє нам зручно розгортати програмне забезпечення для роботичних пристроїв. Завдяки ROS, досить легко підключити новий тип пристрою до мережі. Активна спільнота ROS розробила сотні готових пакетів для різнихроботи та пристрої Інтернету речей протягом років.

Три основні абстракції в середовищі ROS - це вузли, теми та повідомлення. Вузол - це процес / програма, яка виконує певне завдання та спілкується з іншими вузлами через теми. Вузол підписується на тему, очікуючи повідомлень, або публікує власні повідомлення до них. У цій реалізації різні компоненти роботизованої системи можуть взаємодіяти між собою на одному рівні.

На малюнку нижче зображено типовий процес прийняття та виконання завдання робототехнічним пристроєм (промісником) в Robonomics v1, який працює в мережі Ethereum. Блок AIRA сlient - це розгорнуте зображення необхідного програмного забезпечення для пристрою, включаючи ROS. На малюнку показано (у синіх рамках), що пристрій спілкується з децентралізованою мережею через клієнт AIRA, отримуючи та відправляючи повідомлення в відповідні теми ROS. Місія пристрою завершується остаточним відправленням отриманого повідомлення в мережу IPFS, яке містить посилання на дані, що потенційно можуть бути прийняті для монетизації.

!["Процес Robonomics v2"](/blog/images/ocean-protocol-p-2/uml-robonomics-ver1.png)

У Robonomics v2, який розробляється як [парачейн](https://parachain.robonomics.network/) екосистеми Polkadot, процес трохи відрізняється. Наразі доступні функції `launch()` та `datalog()` для керування пристроєм; перша функція дозволяє виконувати транзакції для увімкнення / вимкнення пристрою, а друга відправляє рядок даних на блокчейн. Але журнали та дані пристрою все ще збираються за допомогою тем ROS. Потім їх можна передати безпосередньо до `datalog()`, або спочатку відправити на IPFS.

Отже, першим завданням у процесі інтеграції мережі Ocean Protocol з мережею Robonomics є внесення API протоколу в середовище ROS. Далі, такий пакет для ROS може бути інтегрований в клієнт AIRA та автоматично розгортати створення дататокенів.

Ocean Protocol пропонує кілька варіантів API для роботи з ним, у цьому випадку використовується [ocean.py](https://github.com/oceanprotocol/ocean.py) бібліотека Python.використовується, оскільки ROS має бібліотеку клієнта Python [rospy](http://wiki.ros.org/rospy). Основною метою пакету є створення трьох вузлів ROS, через які роботизоване пристрій може розгортати та створювати монети даних, створювати пул для розміщення на ринку або купувати монети даних для використання. ### Налаштування мережі Ethereum та вузла

У прикладі використовується постачальник Infura для налаштування підключення. Infura дозволяє вам віддалено використовувати інфраструктуру Ethereum без необхідності запуску повного вузла, що дуже зручно для компактного роботизованого пристрою з обмеженою обчислювальною потужністю.

Зверніть увагу, що для прикладу налаштовується підключення до Rinkeby, тестової мережі Ethereum. Якщо вам потрібно перейти на mainnet або іншу, зверніться до [документації Ocean](https://docs.oceanprotocol.com/concepts/networks/).

Кроки для налаштування наступні:

* Перейдіть на https://infura.io та зареєструйтеся.
* На сайті Infura створіть новий проект.
* На сторінці налаштувань проекту запишіть значення вашого ідентифікатора проекту Infura. Це буде використано на наступному кроці.
* Створіть файл `config.ini` у каталозі конфігурації (приклад конфігурації можна знайти у файлі `config_template.ini`).
* Запишіть ваш ідентифікатор проекту Infura для мережі в файлі конфігурації.

### Створення дататокену

Щоб створити дататокен, вам потрібно опублікувати повідомлення на тему `/ocean/create_datatoken`. Повідомлення повинно містити шлях до файлу приватного ключа з вашого облікового запису Metamask, IPFS URL для даних з пристрою, OCEAN та ліквідність дататокену. Приклад повідомлення:

<RbCode>

```sh
rostopic pub /ocean/create_datatoken ocean_ros/Metadata "private_key_path: '/home/user/private_ocean'                     
data_created: '2021-12-28T10:55:11Z'
type: 'dataset' 
name: 'test_ros'
author: 'author'             
license: 'CC0: Public Domain'    
files_content: 'content'
files_content_type: ['text/text']
files_url: ['https://gateway.ipfs.io/ipfs/QmWRjFcQi4Xcisqi8FP3AbGS3PB3gNHgtnfzbcpodKKCBP']                 
tokens_nomber: 100.0
ocean_amount: 10.0
```

</RbCode>

Відповідний вузол ROS створить дататокен, який стане доступним для продажу. Інформацію про дататокен можна побачити в темі `/ocean/token_response`:

<RbCode>

```sh
rostopic echo /ocean/token_response
```

</RbCode>

</TextCard>

### Покупка дататокену

Для покупки дататокену потрібно опублікувати повідомлення в тему `/ocean/get_datatoken`. Повідомлення повинно містити шлях до файлу з приватним ключем з вашого облікового запису Metamask та інформацію про дататокен, який ви хочете придбати (ви можете знайти його на [ринку](https://market.oceanprotocol.com/) або в темі `/ocean/token_response`). Приклад повідомлення (не забудьте змінити `destination` у повідомленні):


<RbCode>

```sh
rostopic pub /ocean/get_datatoken ocean_ros/BuyDatatoken "{private_key_path: '/home/user/private_ocean', destination: '/home/user/', token_address: '0x9fb21F68257F1d718d764B68b1430B6460796e42', did: 'did:op:9fb21F68257F1d718d764B68b1430B6460796e42', pool_address: '0xcF295B85ef5ADd0E513B789477C6d14eA6Bc718a'}"
```

</RbCode>

Шлях до завантаженого файлу даних можна побачити в темі `/ocean/buying_response`:

<RbCode>

```sh
rostopic echo /ocean/buying_response
```

</RbCode>

## Сценарії використання

У цьому розділі будуть описані потенційні приклади застосування автоматичної токенізації даних, отриманих від роботизованих пристроїв.

### Дані про навколишнє середовище

Можливо, найцікавішим прикладом застосування буде токенізація даних про навколишнє середовище. Вартість таких даних лише зростає, враховуючиз урахуванням зростаючого інтересу населення, компаній та регуляторів до проблем управління навколишнім середовищем. Основні проблеми тут - масштаб екології (повітря, вода, ґрунт, помножені на величезні і часто недоступні території Землі) та питання довіри до зібраних даних. Мобільний автоматичний пристрій (дрон для повітря або води) або мережа стаціонарних датчиків ідеально вирішують першу проблему, а блокчейн та децентралізовані технології пом'якшують другу. З можливістю токенізації цих даних за допомогою Ocean Protocol відкриваються зовсім нові ринки, де ідеї, такі як громадянська наука та вуглецеві кредити, ідеально вписуються.

Robonomics працює з пристроями для екології вже довгий час. Наприклад, був розроблений пакет [sensors-connectivity](https://github.com/airalab/sensors-connectivity), який дозволяє перетворити сенсори, сумісні з ROS, на телеметричних агентів. Інженери Robonomics розробляють різні моделі [водних дронів](https://blog.aira.life/water-drone-2-0-first-stage-of-development-db8ad6c81d3c) для збору хімічних показників у річках та озерах, а також досліджують можливість використання [БПЛА](http://www.ijeetc.com/uploadfile/2019/1009/20191009035041545.pdf) для оцінки якості повітря.

<RbCode>

```
1556199372.384576 <=>\x86##5E1567057C105409#DOV01SW#0#WT:6.53#PH:8.89#DO:90.4#COND:348.9#
```

</RbCode>

Приклад зібраних даних про воду: температура, значення рН, розчинений кисень, провідність
У кожному випадку пристрої генерують журнали сенсорів та GPS, які потім [публікуються](https://gateway.ipfs.io/ipfs/QmWRjFcQi4Xcisqi8FP3AbGS3PB3gNHgtnfzbcpodKKCBP) на IPFS та надаються пристроєм в результаті. Тут ми можемо додати цінність цим даним, створивши дататокен за допомогою Ocean Protocol. Мережа Robonomics забезпечить, що дані були зібрані конкретним роботом для конкретного завдання, а Ocean Protocol гарантуватиме, що лише ті, хто володіють дататокеном, матимуть до них доступ.

### Маркетингові дані

Тенденція робототехніки в повсякденному житті зростає, і досить легко знайти приклади роботів-пропагандистів або роботів для клієнтаобслуговування. Досвід взаємодії таких роботів з людьми може бути використаний для маркетингових досліджень, схожих на сайти, які збирають дані про користувачів. Більше того, цінність цих даних буде ще вищою, оскільки взаємодія з людиною відбувається на ближчому рівні.

Ми можемо уявити роботизований пристрій, який, в процесі виконання своєї конкретної задачі (наприклад, робот-офіціант), одночасно збирає візуальну, звукову або іншу інформацію про клієнта. У цьому випадку для створення зовсім нового типу даних про реакцію клієнта на певний вид обслуговування або продукт можна використовувати інструменти машинного навчання. Це відмінна можливість використання даних-токенів як для сирових даних від роботів, так і для тих, що проаналізовані алгоритмами. Більше того, у Ocean Protocol є функція "Обчислення-до-даних", яка дозволяє нам надавати конфіденційні дані для аналітики без порушення конфіденційності.

### Технічні журнали

Покращення алгоритмів роботів також потребує аналізу їх робочих даних. Це стосується, насамперед, машинного навчання, яке потребує наборів даних, але ми також можемо говорити про обмін тривимірними картами (наприклад, для одночасної локалізації та картографування) між пристроями, які працюють в однаковому або схожому середовищі. Монетизація таких технічних журналів буде більш специфічною за галуззю, але цінність полягатиме в їх унікальності та рідкісності.

До прикладів входить токенізація даних, що генеруються роботизованими пилососами для покращення їх продуктивності (обхід перешкод, краще визнання середовища в квартирах). Іншим прикладом є сервісні дані лінії виробництва, обробка яких дозволить оптимізувати витрати на експлуатацію або виробництво.

## Висновок

Приклади показують, що поєднання мережі Robonomics та Ocean Protocol створює зовсім новий тип активів, раніше недоступних через погану автоматизацію технічних процесів та неможливість безпечного контролю та обміну їх результатами. Більше того, це лише очевидні ідеї, і в майбутньому, з поширенням Інтернету речей та роботів, можуть з'явитися зовсім нові ринки для кібер-фізичних даних.
