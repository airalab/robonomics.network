## Kusama OpenGov Causes cₕₐₒₛ

Долгожданная появление новой системы управления, OpenGov (Gov2), наконец, произошла на Kusama, но она принесла с собой много хаоса. Внедрение новой системы OpenGov привело к тому, что ранее запланированные аукционы не были успешно проведены. Сообщество Kusama начало [референдум Gov1 #252](https://kusama.polkassembly.io/referendum/252) для исправления расписания аукционов, но он не был успешно проведен из-за ограничений привилегии AuctionAdmin. Эта ошибка была вызвана тем, что в новой системе управления необходимо представить предложение OpenGov на треке аукционов для запуска аукционов. Было решено настроить параметры OpenGov с помощью [обновления времени выполнения](https://kusama.polkassembly.io/referendum/254), но и этот референдум также не был успешен из-за конфликта между логикой Gov1 и Gov2.

В общем, сообществу Kusama и разработчикам потребовалось пару недель, чтобы разобраться в тонкостях новой системы управления. Это было непредвиденным и повлияло на планы проекта Robonomics по участию в аукционе слотов парачейна в нашей кампании по сбору средств в ноябре 2022 года. Мы уже отменили [кампанию по сбору средств ноября 2022 года](/blog/robonomics-crowdloan-campaigns-november-2022/), которая была начата наparaID 2237 и весь KSM были возвращены участникам краудлона.

## Новые вызовы для продления аренды слота

**Стандартный метод** для обмена аренды парачейн до выпуска OpenGov был следующим:

1. Создать новый paraID с использованием экструзии регистратора (Резерв).
2. Установить on-chain ID для нового paraID.
3. Открыть краудлоан с использованием экструзии краудлоана (Создать).
4. Сделать ставку на аукцион слотов парачейн.
5. В случае успешного завершения аукциона разработчики инициируют действие управления для обмена аренды с этого донорского paraID на рабочий paraID парачейна с использованием функций forceLease и registrar:swap, аналогично предыдущим референдумам: [Обмен аренды Karura](https://kusama.polkassembly.io/referendum/192), [Обмен аренды Moonriver и Khala](https://kusama.polkassembly.io/motion/468), [Обмен аренды Shiden](https://kusama.polkassembly.io/referendum/199), [Обмен аренды Parallel-Heiko](https://kusama.polkassembly.io/referendum/219), [Обмен аренды Genshiro](https://kusama.polkassembly.io/referendum/246).
6. В результате рабочий парачейн получит продленную аренду, продолжит производить блоки и не столкнется с простоем.

Однако с введением OpenGov пороги для действий управления на Kusama стали чрезвычайно высокими. В общих чертах новая система OpenGov имеет несколько треков. Каждый трек может поддерживать определенный тип функции (может вызывать только определенные экструзии). Некоторые треки имеют ограниченную мощность, например, могут давать малые суммы KSM, но другие имеют огромную мощность, такую как Root track.

Пороги для треков OpenGov чрезвычайно высоки, и уже существуют действия в сообществе Kusama по их снижению, так как они уже вызывают проблемы для проектов парачейн, таких как [Kintsugi](https://twitter.com/kintsugi_btc/status/1599036329399943168). Простыми словами, конкретный референдум должен получить достаточную поддержку от заинтересованных лиц сети (владельцев KSM), чтобы референдум перешел из периода голосования в период подтверждения.

Для того чтобы парачейн мог выполнить вышеупомянутый 6-шаговый план по обмену аренды, необходимо снова запустить расписание аукциона парачейна, этоозначает, что референдум OpenGov должен быть принят, чтобы перезапустить аукционы либо через дорожку администрирования аренды, либо возможно в рамках обновления времени выполнения через дорожки Root/Whitelisted Caller. Здесь возникает проблема – для того чтобы референдум был внесен в период подтверждения вовремя, требуется значительный процент всех циркулирующих токенов KSM, проголосовавших в пользу референдума. Типичная явка избирателей на Kusama относительно низкая, что означает, что потребуется, возможно, несколько недель, чтобы референдум дорожки администрирования аренды достиг порога, когда референдум реально может быть проведен в период подтверждения.

Хотя мы все еще надеемся, что сообщество отрегулирует параметры OpenGov, к счастью, существует способ, с помощью которого парачейн может использовать XCM для выполнения 6-шагового плана без необходимости использовать новую систему OpenGov ретрансляционной цепи.

Мы можем зарегистрировать новый paraID, используя внешний резервный регистратор (Reserve), и загрузить оболочку времени выполнения на этот paraID с помощью внешнего регистратора (Register). Этот парачейн должен инициировать сообщения XCM только для обмена аренды. Затем мы отправляем сообщение XCM, содержащее запрос на обмен аренды, от основного парачейна к оболочке парачейна. Когда оболочка парачейна выигрывает аукцион, и аренда становится активной, запускается минимальная среда, которая способна отправить сообщение XCM для обмена слота аренды с основным парачейном.
